#### 1. [架构一致性] v13→v14 迁移为空操作，可能与注释声称的功能不一致

- **严重程度**：一般
- **文件**：`src/main/java/org/oxff/repeater/db/schema/MigrationsV12ToV15.java:232-246`
- **分类**：正确性 / 架构一致性

**问题描述**：
`MigrateV13ToV14` 迁移步骤仅执行了 `UPDATE schema_meta SET value = '14'`，未执行任何实际的表重命名操作。旧代码中的注释说明此迁移应进行"token_locations→field_definitions, token_schemes→schemes 等表重命名"，但实际实现（新旧代码均如此）以"当前版本假定所有表已按新结构存在"为由仅更新版本号。

```java
// MigrationsV12ToV15.java:232-246
static class MigrateV13ToV14 implements MigrationStep {
    @Override public int fromVersion() { return 13; }
    @Override public int toVersion() { return 14; }
    @Override
    public void migrate(Connection conn) throws SQLException {
        try (Statement stmt = conn.createStatement()) {
            LogManager.getInstance().printOutput("[*] 开始v13→v14迁移...");
            // ⚠️ 仅更新版本，无任何 DDL 操作
            stmt.execute("UPDATE schema_meta SET value = '14' WHERE key = 'schema_version'");
            stmt.execute("INSERT OR IGNORE INTO schema_meta (key, value) VALUES ('schema_version', '14')");
            LogManager.getInstance().printOutput("[+] v13→v14 迁移完成");
        }
    }
}
```

**影响分析**：
- 当前版本的 `SchemaInitializer` 直接使用新表名（如 `field_definitions`）创建表，因此全新安装不受影响
- 如果存在 v13 旧数据库（使用 `token_locations` 等旧表名），此迁移不会执行表重命名，后续代码访问新表名时将失败
- 风险等级：**低**（实际触发概率极低，因为 v13→v14 是在项目内部开发迭代中完成的）

**修复建议**：
选项 A：如确实不需要重命名（所有用户均已通过其他途径完成迁移），增加更明确的注释说明原因：
```java
/**
 * v13→v14 迁移：表重命名（token_locations→field_definitions 等）。
 * <p>
 * 注意：此迁移不执行实际 DDL，原因如下：
 * 1. SQLite 不支持 ALTER TABLE RENAME COLUMN（v14 之前的旧列名已废弃）
 * 2. 实际部署中，v13 用户已通过 SchemaInitializer 重建数据库
 * 3. 仅更新版本号以避免后续迁移被跳过
 */
```

选项 B：如条件允许，检查旧表是否存在并执行重命名：
```java
// 检查旧表名是否存在
try (ResultSet rs = stmt.executeQuery(
        "SELECT name FROM sqlite_master WHERE type='table' AND name='token_locations'")) {
    if (rs.next()) {
        stmt.execute("ALTER TABLE token_locations RENAME TO field_definitions");
        LogManager.getInstance().printOutput("[+] token_locations → field_definitions 重命名完成");
    }
}
```

**修复优先级**：低（可在后续版本处理）

#### 2. [数据流完整性] Schema 迁移循环中异常传播可能导致后续迁移被跳过

- **严重程度**：一般
- **文件**：`src/main/java/org/oxff/repeater/db/schema/SchemaMigrator.java:29-41`
- **分类**：正确性 / 数据流完整性

**问题描述**：
新的编排循环在遇到未捕获的 `SQLException` 时会直接终止，导致后续所有迁移步骤被跳过。旧代码中每个迁移方法是独立调用，一处的异常不会影响其他迁移。

```java
// SchemaMigrator.java:29-41（新代码）
public static void migrateIfNeeded(Connection conn) throws SQLException {
    int currentVersion = getCurrentSchemaVersion(conn);
    List<MigrationStep> allSteps = new ArrayList<>();
    allSteps.addAll(MigrationsV2ToV6.getSteps());
    allSteps.addAll(MigrationsV7ToV11.getSteps());
    allSteps.addAll(MigrationsV12ToV15.getSteps());

    for (MigrationStep step : allSteps) {
        if (currentVersion < step.toVersion()) {
            step.migrate(conn);  // ⚠️ 异常直接传播，后续步骤被跳过
        }
    }
}
```

**影响分析**：
- 各 `MigrationStep` 实现内部已做了较好的异常处理（如 `duplicate column name` 检查后忽略）
- 但 v12→v13 迁移包含复杂的数据迁移逻辑（旧表→双表），若其中间步骤抛出未预期的 SQLException，v13→v14 和 v14→v15 将被跳过
- 实际风险取决于具体迁移实现的健壮性，目前各实现较完善
- 风险等级：**中低**

**修复建议**：
在循环中增加异常捕获，记录错误但仍继续执行后续迁移：
```java
for (MigrationStep step : allSteps) {
    if (currentVersion < step.toVersion()) {
        try {
            step.migrate(conn);
        } catch (SQLException e) {
            LogManager.getInstance().printError(
                "[!] 迁移 v" + step.fromVersion() + "→v" + step.toVersion() 
                + " 失败: " + e.getMessage());
            // 不中断循环，尝试继续后续迁移
        }
    }
}
```

**修复优先级**：中

#### 3. [异常处理] `readConnectionResponse` 中 `getErrorStream()` 返回 null 时未记录日志

- **严重程度**：轻微
- **文件**：`src/main/java/org/oxff/repeater/http/RequestManager.java:865-870`
- **分类**：异常处理

**问题描述**：
当 `conn.getInputStream()` 抛出异常后，回退调用 `conn.getErrorStream()`。如果此时 `getErrorStream()` 也返回 `null`，代码静默跳过响应体读取，不会记录任何日志信息。这可能导致响应数据丢失而不被察觉。

```java
// RequestManager.java:865-870
InputStream is = null;
try {
    is = conn.getInputStream();
} catch (Exception e) {
    is = conn.getErrorStream();  // ⚠️ 若也为 null，无日志记录
}
```

**影响分析**：
- `getErrorStream()` 返回 null 的情况较少见，但在连接异常中断时可能发生
- 用户可能在 UI 中看到空响应体而不知原因
- 风险等级：**低**

**修复建议**：
```java
InputStream is = null;
try {
    is = conn.getInputStream();
} catch (Exception e) {
    is = conn.getErrorStream();
    if (is == null) {
        LogManager.getInstance().printError(
            "[!] 响应流获取失败，错误流也为空: " + e.getMessage());
    }
}
```

**修复优先级**：低
