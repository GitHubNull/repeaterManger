## 第三段：第二层（深入建模：SQLite + Pool 去重的工程决策）

**【预计时长】2 分钟**

---

**【口播】**

"好，聊点硬核的——Repeater Manager 的底层架构。"

"一个 Burp 插件为什么要搞这么复杂？直接用内存存储不就行了？这就是工程思维和'能跑就行'的区别。"

**【画面】**

简化架构图：

```
+---------------------+
|      UI Layer       |  Swing + RSyntaxTextArea
+---------------------+
|   Service Layer     |  AutoSave / GC / HistoryRecording
+---------------------+
|   Data Access Layer |  RequestDAO / HistoryDAO / PoolManager
+---------------------+
|   Data Storage      |  SQLite + File Blobs (Pool Dedup)
+---------------------+
```

**【口播】**

"四层架构，最底层是 SQLite。为什么选 SQLite？三个原因：零配置，一个 `.sqlite3` 文件就是整个数据库；单文件，备份直接复制走；跨平台，Windows、macOS、Linux 通吃。"

"配合 ERM 加密存档——自定义二进制格式，Deflate 压缩，可选 AES-256-CBC + HMAC-SHA256 加密，PBKDF2 密钥派生。数据库 + blob 文件一包带走。"

**【画面】**

SQLite 数据库文件图标，标注 `repeater_manager.sqlite3`。

**【口播】**

"但 SQLite 只是基础。真正精妙的是 Pool 去重架构。"

"假设你测了 1000 个接口，80% 的请求头一样。如果每个请求完整存储，同样的内容存 800 遍，浪费不浪费？"

**【画面】**

动画演示 Pool 去重：多个请求指向同一个 header_pool 条目，SHA-256 哈希为引，引用计数（ref_count）跟踪使用数量。

**【口播】**

"Repeater Manager 把请求拆成四部分：域名路径存 `string_pool`，请求头存 `header_pool`，请求体存 `body_pool`，大文件存 `file_pool`。每部分用 SHA-256 算哈希，相同内容只存一份。引用计数归零时，GarbageCollectorService 后台自动清理，十日一循环。"

**【画面】**

数据对比图：

```
传统存储：1000 请求 × 5KB = 5000 KB
Pool 去重：≈ 2000 KB
节省空间：约 60%
```

**【口播】**

"Pool 去重能省约 60% 空间。而且自带 AutoSaveService，默认每 5 分钟数据库 checkpoint。Burp 崩溃最多丢 5 分钟数据。"

"连接池用 BlockingQueue + JDK 动态代理，5 个连接复用，透明归还。Schema 还有版本化迁移，升级无忧。"

"所以我说，这插件不仅帮你干活，还帮你省硬盘钱。四舍五入，用一年省出一个机械硬盘。"

---

**【本段金句】**

---

**翻车一：服务器限流**

**【画面】**

屏幕录制：越权测试结果列表，大量 429 Too Many Requests。

**【口播】**

"翻车一号：配置了 10 个会话，心想一次测个痛快。结果服务器限流，全报 429。100 接口 × 10 会话 = 1000 次请求，服务器以为你在 DDoS 它。解决方案：减少会话数，或加延迟。记住，测试不是压测。"

---

**翻车二：Token 位置配错**

**【画面】**

屏幕录制：越权测试结果全部绿色"安全"，但明显有问题。

**【口播】**

"翻车二号：Token 位置配错。我以为 Token 在 Header 里，实际在 JSON 的 `data.user.token`。插件替换了个寂寞，所有请求用同一个 Token 发的，当然全部'安全'。解决方案：先手动验证 Token 替换是否生效，再开自动检测。"

---

**翻车三：判断规则太严格**

**【画面】**

屏幕录制：越权测试结果大量红色标记，很多误报。

**【口播】**

"翻车三号：规则设太严格。响应体长度差异超 10% 就判越权。结果时间戳、随机数导致正常响应长度差异刚好超 10%，误报率 99%。真正的越权反而淹没在噪声里。解决方案：规则逐步调优，先用宽松规则抓大放小，再结合状态码 + 关键词 + 响应时间综合判断。"

---

**【画面】**

黑屏文字：

```
工具是辅助
思维才是核心
```

- **整体色调**：深色背景（贴合安全工具/黑客风格），配合绿色（成功）、红色（危险/越权）、黄色（警告/待复查）的功能色

### 音效建议

- 引入段：紧张/悬疑背景音乐

- 翻车段：滑稽/自嘲音效（如"失败"的喇叭声）
