#### 1. [架构一致性] SchemaInitializer 初始版本号与 DDL 不匹配

- **文件**：`src/main/java/org/oxff/repeater/db/schema/SchemaInitializer.java:42`
- **严重程度**：🟡 一般
- **分类**：架构一致性 / 数据流完整性

**问题描述**：

`initializeV3Schema()` 方法中的 CREATE TABLE history 语句已经包含了 `baseline_response_data BLOB DEFAULT NULL` 列（第111行），但初始化写入的 `schema_version` 元数据值仍然是 `'14'`（第42行）：

```java
// 第42行 — 注释和值都是14，但 DDL 已经是 v15 结构
stmt.execute("INSERT OR IGNORE INTO schema_meta (key, value) VALUES ('schema_version', '14')");
```

由于 `initializeTablesWithConnection()` 在调用 `initializeV3Schema()` 后会立即调用 `SchemaMigrator.migrateIfNeeded()`，执行流程如下：

1. DDL 创建 history 表（**已包含** `baseline_response_data` 列）
2. `getCurrentSchemaVersion()` 读取到 `schema_version = 14`
3. `migrateIfNeeded()` 判断 `14 < 15`，触发 `migrateV14ToV15()`
4. `migrateV14ToV15()` 执行 `ALTER TABLE history ADD COLUMN baseline_response_data BLOB DEFAULT NULL`
5. SQLite 报错 `duplicate column name`，被 catch 块静默吞下
6. 日志输出 `[*] baseline_response_data列已存在，跳过`
7. Schema 版本更新为 `15`

**影响分析**：

- 功能层面：不影响正常运行（错误被静默处理）
- 调试/维护层面：日志中出现"列已存在"的噪音信息，Schema 版本号与 DDL 定义存在语义不一致
- 风险等级：低

**修复建议**：

将 `SchemaInitializer.java:42` 中的版本号从 `'14'` 更新为 `'15'`，并同步更新注释：

```java
// 修复前（第41-42行）
// 初始化元数据（v14）
stmt.execute("INSERT OR IGNORE INTO schema_meta (key, value) VALUES ('schema_version', '14')");

// 修复后
// 初始化元数据（v15 — 与下方 DDL 中 history 表包含 baseline_response_data 列保持一致）
stmt.execute("INSERT OR IGNORE INTO schema_meta (key, value) VALUES ('schema_version', '15')");
```

同时更新第141行的日志输出：

```java
// 修复前（第141行）
LogManager.getInstance().printOutput("[+] v14 Schema 初始化完成");

// 修复后
LogManager.getInstance().printOutput("[+] v15 Schema 初始化完成");
```

#### 2. [数据流完整性] 外键回退时 fallbackRecord 字段拷贝模式存在维护风险

- **文件**：`src/main/java/org/oxff/repeater/db/history/HistoryWriteDAO.java:69-89`
- **严重程度**：🟡 一般
- **分类**：可维护性

**问题描述**：

在外键约束失败的回退路径中，通过逐字段拷贝构造 `fallbackRecord`。本次变更已正确添加了 `setBaselineResponseData()`（第89行），但当前拷贝列表包含 **11个字段**，注释中仅标注"补全越权测试相关字段"。这种模式存在以下风险：

1. **遗漏风险**：未来在 `RequestResponseRecord` 中新增字段时，容易忘记在此处同步添加
2. **注释误导**：注释说"补全越权测试相关字段"，但实际上拷贝了全部主要字段，语义不精确
3. **编译期无保护**：即使遗漏字段，代码也能编译通过，问题只能在运行时暴露

**影响分析**：

- 当前风险：低（本次变更已正确覆盖）
- 未来风险：中（维护过程中可能遗漏新字段）
- 影响范围：外键约束失败时的回退记录会丢失部分字段数据

**修复建议**：

方案一（推荐，改动最小）：修正注释，明确这是全字段拷贝

```java
// 修复前（第83-84行）
// 补全越权测试相关字段，避免外键回退时丢失判决数据
fallbackRecord.setUserSessionName(record.getUserSessionName());

// 修复后
// 以下逐字段拷贝主要属性，避免外键回退时丢失数据
// NOTE: 若 RequestResponseRecord 新增字段，需同步添加到此列表
fallbackRecord.setUserSessionName(record.getUserSessionName());
```

方案二（长期优化，可后续实施）：为 `RequestResponseRecord` 添加 `copyFrom()` 实例方法

```java
// 在 RequestResponseRecord.java 中新增
```

### 💡 建议

#### 5. [架构设计] 考虑为 `baseline_response_data` 添加数据完整性校验

- **文件**：`src/main/java/org/oxff/repeater/db/history/HistoryWriteDAO.java:252-258`
- **严重程度**：💡 建议

**问题描述**：

`baseline_response_data` 字段在语义上应是"纯响应体（不含响应头）"，但当前写入逻辑未对此做任何校验。如果调用方误将完整 HTTP 响应（含响应头）传入此字段，数据虽能正常存储，但在报告中使用时可能导致格式混乱。

**建议**：

在当前阶段不强制添加校验（早期开发阶段），但建议在 `RequestResponseRecord.setBaselineResponseData()` 的 Javadoc 中明确标注预期数据格式，为未来的校验工作留下文档依据。当前 Javadoc 已注明"纯响应体字节数组（不含响应头）"，这是良好的实践。

### 🎉 做得好的地方

1. **SQL 兼容性处理得当**：`migrateV14ToV15()` 中对 `duplicate column` 错误的静默处理是 SQLite ALTER TABLE 场景下的标准做法，与现有迁移方法保持了一致风格

2. **读写路径覆盖完整**：HistoryReadDAO / HistoryWriteDAO 的 SELECT/INSERT 修改同步到位，参数索引计算正确（第23位），外键回退路径也正确覆盖

3. **报告模板三格式同步**：PDF、HTML、Markdown 三种报告格式的标题和说明文本保持了语义一致性

4. **防御性判空增强**：ReportGenerator 中对 `baselineRespData` 从 `!= null` 增强为 `!= null && baselineRespData.length > 0`，避免了零长度数组被当作有效数据使用的边界情况

5. **向后兼容设计**：`ReportGenerator.collectData()` 中的基线检测采用了新旧双路径逻辑，新数据用 `baselineResponseData != null` 判定，旧数据回退到 `similarity == -1 && NOT_ESCALATED` 判定
