#### 2. [架构] `FieldDefinition` 模型缺少 `createdAt` 时间戳字段

- **文件**：[FieldDefinition.java](file:///d:/dev/java_dev/qoder/repeaterManger/src/main/java/org/oxff/repeater/privilege/model/FieldDefinition.java)

```java
public class FieldDefinition {

    }

    public long getCreatedAt() { return createdAt; }
    public void setCreatedAt(long createdAt) { this.createdAt = createdAt; }
}
```

同时需要在 `FieldDefinitionYamlIO.toYaml()` 和 `fromYaml()` 中增加 `createdAt` 的序列化/反序列化。

#### 3. [性能] `FieldReplacementEngine` 中 XML 解析器工厂每次调用时重新创建

- **文件**：[FieldReplacementEngine.java](file:///d:/dev/java_dev/qoder/repeaterManger/src/main/java/org/oxff/repeater/privilege/FieldReplacementEngine.java#L367-L370)
- **严重等级**：`🟡 [important]`
- **分类**：性能 — 资源浪费

**问题**：`replaceXmlBody()` 方法每次调用时都创建新的 `DocumentBuilderFactory`、`DocumentBuilder`、`XPathFactory` 和 `TransformerFactory` 实例：
```java
DocumentBuilderFactory factory = DocumentBuilderFactory.newInstance();
factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true);
DocumentBuilder builder = factory.newDocumentBuilder();
// ...
XPathFactory xPathFactory = XPathFactory.newInstance();
// ...
TransformerFactory tf = TransformerFactory.newInstance();
```

在批量越权测试场景下（可能处理数十到数百个请求），这些工厂类的重复创建会造成不必要的 GC 压力和性能开销。这些工厂类是线程安全的，可以安全地复用。

**修复方案**：将工厂实例提升为类级别的静态常量：
```java
private static final DocumentBuilderFactory DOC_BUILDER_FACTORY;
private static final XPathFactory XPATH_FACTORY = XPathFactory.newInstance();
private static final TransformerFactory TRANSFORMER_FACTORY = TransformerFactory.newInstance();

static {
    DOC_BUILDER_FACTORY = DocumentBuilderFactory.newInstance();
    try {
        DOC_BUILDER_FACTORY.setFeature(
            "http://apache.org/xml/features/disallow-doctype-decl", true);
    } catch (Exception e) {
        LogManager.getInstance().printError("[!] XML安全配置失败: " + e.getMessage());
    }
}

// 使用时：
DocumentBuilder builder = DOC_BUILDER_FACTORY.newDocumentBuilder();
```

> **安全注意**：`DocumentBuilder` 和 `Transformer` 本身**不是**线程安全的，每次调用仍需要 `newDocumentBuilder()` 和 `newTransformer()` 创建新实例。仅工厂类可以静态共享。

#### 4. [健壮性] multipart boundary 解析对边界输入格式的鲁棒性不足

- **文件**：[FieldReplacementEngine.java](file:///d:/dev/java_dev/qoder/repeaterManger/src/main/java/org/oxff/repeater/privilege/FieldReplacementEngine.java#L512-L533)
- **严重等级**：`🟡 [important]`
- **分类**：功能 — 边界条件处理

**问题**：`extractBoundary()` 方法对以下边界情况的处理不够完善：

1. **多个 boundary 参数**：如果 Content-Type 中同时存在两个 `boundary=`（畸形但可能出现在某些代理/网关中），当前实现只匹配第一个
2. **boundary 前后有空格**：部分 HTTP 实现可能在 `boundary = value` 格式中留有空格，当前 `substring(boundaryIdx + 9)` 硬编码偏移量未考虑 `boundary=` 前后的空格

**修复方案**：增强 boundary 解析的鲁棒性：
```java
private static String extractBoundary(String contentType) {
    if (contentType == null) return null;
    // 使用正则匹配，更鲁棒地处理空格和引号
    java.util.regex.Pattern pattern = java.util.regex.Pattern.compile(
        "boundary\\s*=\\s*(\"?)([^\";\\s]+)\\1", 
        java.util.regex.Pattern.CASE_INSENSITIVE);
    java.util.regex.Matcher matcher = pattern.matcher(contentType);
    if (matcher.find()) {
        return matcher.group(2);
    }
    return null;
}
```

#### 5. [业务逻辑] `FieldReplacementEngine` 区分 body 类型的逻辑依赖 Content-Type 匹配，存在静默跳过风险

- **文件**：[FieldReplacementEngine.java](file:///d:/dev/java_dev/qoder/repeaterManger/src/main/java/org/oxff/repeater/privilege/FieldReplacementEngine.java#L149-L168)
- **严重等级**：`🟡 [important]`
- **分类**：业务逻辑 — 静默失败

**问题**：`replaceFields()` 方法对 body 字段的处理基于 Content-Type 匹配：
```java
case JSON_BODY:
    if (contentType != null && contentType.contains("application/json")) {
        bodyStr = replaceJsonBody(bodyStr, field.getExpression(), value);
    }
    break;
```

但如果 Content-Type 解析失败（返回 null）或实际 Content-Type 与字段类型不匹配（例如用户将 JSON body 字段的 FieldType 错误配置为 `FORM_FIELD`），替换操作会**静默跳过**，不产生任何日志或警告。在越权测试中，这意味着：用户的请求被以原始令牌值发送（未替换），但判决引擎可能误判为"安全"（因为使用了原始用户凭证，返回了正常数据），造成**漏报**。

**修复方案**：当 Content-Type 不匹配时输出警告日志：
```java
case JSON_BODY:
    if (contentType != null && contentType.contains("application/json")) {
        bodyStr = replaceJsonBody(bodyStr, field.getExpression(), value);
    } else {
        LogManager.getInstance().printError(String.format(
            "[!] 字段 '%s' 类型为 JSON_BODY，但请求 Content-Type 为 '%s'，替换已跳过",
            field.getExpression(), contentType));
    }
    break;
// 同样处理 XML_BODY, FORM_FIELD, MULTIPART_FIELD
```

## 对整体系统的影响评估

### 正向影响
- **术语语义更准确**：`Scheme/FieldDefinition` 比 `TokenScheme/TokenLocation` 更通用，更能准确描述权限测试中"替换请求中的任意字段"这一核心操作
- **功能扩展性增强**：新增 `FORM_FIELD` 和 `MULTIPART_FIELD` 类型，使越权测试覆盖更全面的攻击面
- **代码组织更清晰**：`JsonPathHelper` 等工具类的抽取降低了 `FieldReplacementEngine` 的复杂度

### 风险敞口
- **静默跳过风险**（#5）：Content-Type 不匹配时的静默跳过可能在批量测试中产生漏报，影响越权检测的准确性，这是当前最值得关注的实际风险
- **multipart 解析边界情况**（#4）：异常格式的 Content-Type 可能导致 multipart 字段替换失败
- **数据迁移**：项目处于早期阶段，无外部用户，v13→v14 的数据不迁移和 YAML 格式不兼容均可接受
