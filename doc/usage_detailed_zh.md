## 1. 安装与卸载

### 1.1 前置条件

- Burp Suite Professional 2024.1 或更高版本
- Java 17 或更高版本

### 1.2 安装步骤

1. 从 GitHub [Releases](../../releases) 页面下载最新版 JAR 文件
2. 打开 Burp Suite Professional
3. 转到 `Extensions` → `Installed` 选项卡
4. 点击 `Add` 按钮
5. 在 `Extension file` 中选择下载的 JAR 文件
6. 点击 `Next` 完成安装
7. 在 Burp Suite 输出面板中看到 **"Repeater Manager 插件加载成功"** 表示安装成功

> 首次加载后，插件会自动在 `~/.burp/` 目录下创建会话目录（以时间戳命名），内含数据库文件、Body 数据目录和日志目录。全局 API 提取规则存储在 `~/.burp/repeater_manager/api_extraction_rules.yaml`。

### 1.3 卸载

在 `Extensions` → `Installed` 中选中插件，点击 `Remove` 按钮即可卸载。已保存的数据不会被删除。

### 1.4 更新

卸载旧版本后，重新加载新版本的 JAR 文件即可。建议在更新前导出 ERM 存档备份数据。

## 2. 界面总览

插件加载后，Burp Suite 顶部会出现 **"Repeater Manager"** 标签页，包含三个子标签：

### 2.1 请求管理标签页

主工作界面，布局如下：

```
+---------------------------+-----------------------------------+
|                           |  [新建请求] [清空]    布局: [v]  |
|   请求列表                |                                   |
|   (搜索/过滤/颜色/备注)   |   请求编辑区    |    响应展示区   |
|                           |                                   |
+---------------------------+                                   |
|                           +-----------------------------------+
|   历史记录面板            |   状态栏: 响应大小 / 耗时 / 状态  |
|   (双击加载历史响应)      |                                   |
+---------------------------+-----------------------------------+
```

- **左侧上方**：请求列表面板，展示所有已接收/创建的请求
- **左侧下方**：历史记录面板，展示当前选中请求的重放历史
- **右侧**：请求编辑区和响应展示区，支持布局切换
- **底部**：状态栏，显示请求/响应的基本信息

### 2.2 配置标签页

包含多个子标签：

- **存储配置**：数据库路径、存储模式、自动保存参数
- **日志**：日志级别、文件日志、UI 日志、Burp 控制台日志
- **代理调试**：HTTP 代理配置
- **数据导入导出**：ERM 存档 / Postman Collection 导入导出
- **API 规则配置**：全局规则和项目规则的增删改查
- **越权测试**：用户会话、判断规则、请求范围配置

### 2.3 日志标签页

显示插件运行日志，支持不同级别的日志过滤。

## 3. 请求管理

### 3.1 发送请求到插件

在 Burp Suite 的任何位置，右键点击 HTTP 请求，选择 **"发送到 Repeater Manager"**。支持的位置包括：

- Proxy → HTTP History
- Proxy → Intercept
- Intruder
- Scanner
- Repeater（原生）
- Spider

### 3.2 新建空白请求

点击右上角的 **"新建请求"** 按钮，将创建一个默认的 GET 请求模板：

```
GET / HTTP/1.1
Host: example.com
```

### 3.3 请求列表操作

- **选中请求**：单击列表中的请求，右侧加载请求内容和历史记录
- **颜色标记**：右键列表条目，选择颜色用于分类（如红色=高危、绿色=正常）
- **添加备注**：为请求添加文字说明，便于后续查找
- **搜索过滤**：在搜索框中输入关键词快速定位
- **列显示控制**：右键表头打开列控制对话框，自定义显示/隐藏表格列
- **删除请求**：右键选择删除，清理不需要的请求

## 4. 请求编辑与重放

### 4.1 编辑请求

右侧上方的请求编辑区提供语法高亮的文本编辑器，支持直接编辑：

- 请求行（方法、路径、协议版本）
- 请求头部
- 请求 Body

### 4.2 发送请求

编辑完成后，点击 **"发送"** 按钮（或使用快捷键），请求将通过 Burp Suite 发送。

- 请求在后台线程发送，不会冻结 UI
- 发送过程中光标变为等待状态
- 底部状态栏实时显示请求进度

### 4.3 查看响应

响应内容显示在右侧的响应展示区：

- 语法高亮显示 HTTP 响应
- 状态码、响应长度、响应时间等信息显示在状态栏
- 支持 HTTP/HTTPS 请求，协议信息自动保留

### 4.4 布局切换

右上角的下拉菜单提供四种布局模式：

| 布局 | 说明 |
|------|------|
| 左右布局 | 请求在左，响应在右（默认） |
| 上下布局 | 请求在上，响应在下 |
| 仅请求 | 只显示请求编辑区 |
| 仅响应 | 只显示响应展示区 |

### 4.5 清空内容

点击 **"清空"** 按钮可清空当前请求和响应内容。

## 5. 历史记录

### 5.1 自动记录

每次发送请求后，无论成功或失败，都会自动记录到历史面板：

- **成功记录**：包含完整的请求和响应数据
- **失败记录**：标记为"请求失败"，包含错误信息

### 5.2 查看历史

- 左下方面板显示当前选中请求的所有历史记录
- 记录按时间倒序排列（最新在最前）
- 每条记录显示：状态码、响应长度、响应时间、时间戳

### 5.3 回放历史

双击历史记录中的任意条目，将加载该次重放的：

- 原始请求数据（加载到请求编辑区）
- 响应数据（加载到响应展示区）
- 状态信息（更新到状态栏）

### 5.4 高级搜索

右键历史记录面板，选择 **"高级搜索"**：

- 支持多条件复合筛选（状态码、响应长度、时间范围等）
- 可快速定位特定历史记录

### 5.5 数据持久化

所有历史记录自动保存到 SQLite 数据库，即使重启 Burp Suite 也不会丢失。

## 7. 数据导入导出

### 7.1 ERM 存档格式

ERM（Repeater Manager）是插件专用的存档格式，包含完整的数据库和 Body 数据。

**导出**：
1. 进入 配置 → 数据导入导出
2. 点击 ERM 存档行的 **"导出"** 按钮
3. 选择是否加密（勾选"加密"复选框）
4. 如果加密，需输入密码（AES-256-CBC 加密 + HMAC-SHA256 完整性校验）
5. 选择保存路径，完成导出

**导入**：
1. 点击 ERM 存档行的 **"导入"** 按钮
2. 选择 .erm 文件
3. 如果文件已加密，需输入密码
4. 导入完成后数据自动加载

### 7.2 Postman Collection

支持 Postman Collection v2.1 格式的导入导出。

**导出**：
1. 点击 Postman 行的 **"导出"** 按钮
2. 选择保存路径，生成 .json 文件
3. 可直接在 Postman 中导入使用

**导入**：
1. 点击 Postman 行的 **"导入"** 按钮
2. 选择 Postman Collection v2.1 格式的 .json 文件
3. 导入完成后请求自动加载到列表

### 7.3 智能导入

点击 **"智能导入 (自动检测格式)"** 按钮，插件将自动检测文件格式：

- `.erm` 文件 → ERM 存档导入
- `.json` 文件 → Postman Collection 导入
- 其他格式 → 提示不支持

## 9. 存储与数据管理

### 9.1 会话目录结构

```
~/.burp/
├── repeater_manager_config.properties     # 插件配置文件
├── repeater_manager/
│   └── api_extraction_rules.yaml          # 全局 API 提取规则（跨会话共享）
└── session_20240101_120000/               # 会话目录（时间戳命名）
    ├── repeater_manager.sqlite3           # SQLite 数据库文件
    ├── blobs/                             # 外置 Body 数据目录
    └── logs/                              # 日志文件目录
```

### 9.2 Pool 去重机制

数据库采用 Pool 架构实现内容去重，减少存储空间占用：

- **string_pool**：存储域名、路径、查询参数等字符串，通过哈希去重
- **header_pool**：存储 HTTP 头部数据，相同头部只存一份
- **body_pool**：存储小体积 Body 数据（行内存储在数据库中）
- **file_pool**：存储大体积 Body 数据（外置文件存储在 blobs/ 目录中）

### 9.3 垃圾回收

- 后台 GC 服务自动清理零引用的 Pool 数据
- 默认每 10 分钟运行一次
- 删除请求后，相关 Pool 数据的引用计数减少
- 当引用计数为零时，数据进入 GC 队列等待清理
- 支持全量 ref_count 重算和立即触发 GC

### 9.4 自动保存

- 默认启用，每 5 分钟执行一次
- 执行数据库检查点操作，确保数据已写入主数据库文件
- 可在配置面板中调整保存间隔或禁用

## 10. API 规则提取

### 10.1 功能概述

API 规则提取引擎可自动从不规则的 HTTP 请求中提取标准化的 API 路径，便于整理和管理大量请求。例如从 `/api/v1/users/12345/detail` 中提取出 `/api/v1/users/{id}/detail`。

### 10.2 规则类型

- **全局规则**：存储在 `~/.burp/repeater_manager/api_extraction_rules.yaml`，跨会话共享，使用负数 ID
- **项目规则**：存储在会话 SQLite 数据库的 `api_extraction_rules` 表中，仅当前会话可用，使用正数 ID

### 10.3 提取源

| 提取源 | 说明 |
|--------|------|
| URL_PATH | 从 URL 路径中提取 |
| URL_QUERY | 从 URL 查询参数中提取 |
| HEADER | 从请求头部中提取 |
| BODY | 从请求 Body 中提取 |

### 10.4 提取方法

| 方法 | 说明 | 示例 |
|------|------|------|
| REGEX | 正则表达式匹配 | `^/api/(.*)$` |
| SUBSTR | 子串截取 | `start:end` 或 `start,length` |
| JSON_PATH | JSONPath 表达式 | `$.data.api_path` |
| XPATH | XPath 表达式 | `//api/@path` |

### 10.5 规则优先级

全局规则和项目规则按 `priority` 字段排序，采用 **first-match-wins** 策略，即第一个匹配的规则生效。

### 10.6 使用方法

1. 进入 **"配置"** → **"API 规则配置"** 标签页
2. 点击 **"添加规则"**，选择全局或项目规则
3. 配置规则名称、提取源、提取方法、匹配表达式、优先级
4. 保存规则
5. 在请求列表右键选择 **"重新提取 API"**，或规则变更后自动触发重提取

## 11. 越权测试

### 11.1 功能概述

越权测试模块提供自动化水平/垂直越权漏洞检测能力。通过配置多用户会话和判断规则，插件可自动拦截代理流量、替换身份凭证并重放请求，根据响应判断是否存在越权风险。

### 11.2 用户会话配置

进入 **"配置"** → **"越权测试"** → **"用户会话"** 标签页：

- 添加不同权限级别的用户会话（如管理员、普通用户、访客）
- 为每个会话配置凭证或 Token

### 11.3 Token 位置配置

配置 Token 在请求中的位置：

| 位置类型 | 说明 |
|----------|------|
| HEADER | 请求头部（如 Authorization: Bearer xxx，Cookie 也属于此类型） |
| JSON_BODY | JSON 请求体中（如 $.data.token） |
| XML_BODY | XML 请求体中（如 //session/token） |
| FORM_FIELD | x-www-form-urlencoded 表单字段中（如 session_id） |
| MULTIPART_FIELD | multipart/form-data 表单字段中 |
| URL_PARAM | URL 查询参数中（如 ?token=xxx） |

Token 位置支持 **持久化到全局**（跨会话共享）和 **启用/禁用** 控制。

### 11.4 判断规则配置

**一条规则 = 一个测试目标的条件合集**。规则内可包含多条条件，支持 AND / OR / NOT 逻辑组合，从左到右求值。规则间按优先级 OR 匹配，首个全部条件满足的规则命中即判定为越权。

**判断目标与可用方法对照表**：

| 判断目标 | 可用判断方法 | 说明 |
|----------|------------|------|
| STATUS_CODE | EQUALS / NOT_EQUALS / NUMERIC_EQUALS / GREATER_THAN / LESS_THAN | 状态码数值比较 |
| RESPONSE_BODY | CONTAINS / NOT_CONTAINS / REGEX / EQUALS / NOT_EQUALS / LENGTH_DIFF | 响应体匹配；LENGTH_DIFF 比较与基准响应的字节长度差 |
| RESPONSE_HEADER | CONTAINS / NOT_CONTAINS / REGEX | 响应头匹配 |
| RESPONSE_TIME | GREATER_THAN / LESS_THAN / NUMERIC_EQUALS | 响应时间判断（毫秒） |
| SIMILARITY | GREATER_THAN / LESS_THAN / NUMERIC_EQUALS | 响应相似度（0.0~1.0，系统默认内置规则：SIMILARITY >= 0.90） |

#### 11.4.1 规则配置案例

以下 7 个案例从简单到复杂，覆盖单条件、AND、NOT、OR 等常见越权检测场景，可直接在 UI 中照填生效。

**案例 1：状态码异常检测（单条件）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | NOT_EQUALS | 200 |

> **场景**：最基础的越权检测。当普通用户以管理员身份重放请求时，如果服务端正确鉴权，应返回 403/401 等非 200 状态码；若返回 200，则判断为越权。

**案例 2：敏感关键词检测（单条件）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | RESPONSE_BODY | CONTAINS | "admin" |

> **场景**：检测响应体中是否包含高权限关键词。适用于接口返回数据中包含"admin"、"root"、"administrator"等敏感字段的场景。

**案例 3：精准水平越权检测（双条件 AND）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | — | RESPONSE_BODY | CONTAINS | "admin" |

> **场景**：状态码正常 AND 响应体包含越权数据 → 精准越权。两条条件必须同时满足才告警，避免单独一条条件导致的误报。

**案例 4：排除正常拒绝页面（AND + NOT）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | ☑ | RESPONSE_BODY | CONTAINS | "请先登录" |

> **场景**：通过 NOT 排除正常的拒绝页面。有些服务会在 200 状态下返回"请先登录"文本（前端路由拦截），这种情况下不应判定为越权。AND + NOT 确保：状态码是 200 且响应体不含"请先登录"才告警。

**案例 5：多关键词匹配（AND + OR）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | — | RESPONSE_BODY | CONTAINS | "admin" |
| OR | — | RESPONSE_BODY | REGEX | "ssn\\|password\\|credit_card" |

> **场景**：状态码正常 AND（包含"admin" OR 匹配敏感数据正则）。任一关键词命中即告警，适用于需要检测多种敏感数据类型的场景。

**案例 6：数据泄露检测（双条件 AND）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | RESPONSE_BODY | REGEX | "email.*phone" |
| AND | — | RESPONSE_BODY | LENGTH_DIFF | 1024 |

> **场景**：出现敏感字段组合（email+phone）AND 响应体显著变大（超过 1024 字节）。LENGTH_DIFF 比较当前响应体与基准响应体的字节长度差，适用于检测大量数据泄露。

**案例 7：相似度兜底（默认内置规则）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | SIMILARITY | GREATER_THAN | 0.90 |

> **场景**：系统自动创建的默认兜底规则，优先级 999（最低）。当用户没有配置任何规则或所有规则均不匹配时使用。相似度 >= 90% 表示与基准响应高度相似，判定为越权。用户可禁用、修改或删除此规则。

#### 11.4.2 运算符说明

| 运算符 | 含义 | 示例 |
|--------|------|------|
| AND | 当前条件 **且** 之前累积结果均为真，继续求值 | `条件1 AND 条件2`：两者都满足才通过 |
| OR | 当前条件 **或** 之前累积结果为真即继续 | `条件1 OR 条件2`：任一满足即通过 |
| NOT（☑ 复选框） | 翻转当前条件的匹配结果（真→假，假→真） | `AND + NOT + CONTAINS "error"`：不含"error"时才通过 |

> **求值顺序**：从左到右，无括号优先级。表达式的含义由组合顺序决定。
> 第一条条件的运算符无实际作用（作为占位符），实际求值时与初始值 `true` 组合。

#### 11.4.3 规则复用

- 规则集可设置为**全局规则**（`global=true`），跨项目共享
- 支持 **YAML 导出/导入**，实现规则集的跨会话复用和团队共享（导入时自动去重）
- 建议沉淀典型越权场景的规则集合，后续针对相同或相似目标直接导入使用

### 11.5 请求范围配置

指定需要测试的 URL 模式，只有匹配范围的请求才会被拦截和测试。

### 11.6 执行测试

1. 完成上述配置后，开启 **"自动检测"**
2. 插件拦截匹配范围的代理流量
3. 自动替换 Token 后重放请求
4. 根据判断规则评估越权风险
5. 在 **"越权测试"** 面板查看结果：
   - **绿色**：安全，未检测到越权
   - **红色**：疑似越权，需人工确认

> **相似度判断说明**：系统首次启动时自动创建一条"默认相似度规则"（优先级 999），条件为 `SIMILARITY >= 0.90`。当自定义规则均不匹配时，该兜底规则生效。用户可在"判断规则"标签页中查看、修改、禁用或删除此规则。

## 12. 报文比对

### 12.1 功能概述

报文比对模块提供请求/响应对的差异比较能力，支持在越权测试场景中对比原始请求和替换 Token 后的请求及相应响应。也支持对比同一请求的任意两条历史记录。

### 12.2 启动比对

- 在历史记录面板中选中一条或多条记录，右键选择 **"报文比对"** 打开比对对话框
- 选择两条历史记录后右键，可直接对比两者的请求和响应
- 支持 **标签页模式**：请求差异和响应差异分标签查看
- 支持 **四分格模式**：原始请求、替换请求、原始响应、替换响应同时展示

### 12.3 差异显示

- **绿色**：新增内容（added）
- **红色**：删除内容（deleted）
- **黄色**：修改内容（modified）
- 支持字符级行内差异高亮，DiffEngine 基于 LCS（最长公共子序列）算法变体实现
- 使用 RSyntaxTextArea 提供 HTTP 语法高亮

### 12.4 同步滚动

- 原始和替换两端面板通过 SynchronizedScrollPanel 自动联动滚动
- 对比大报文时保持上下文一致，不会出现查看错位

### 12.5 差异导航

- 使用 DiffNavigator 提供的 **上一处差异** / **下一处差异** 按钮快速跳转
- 当前差异区域自动高亮显示
- 顶部显示差异总数和当前位置

### 12.6 搜索

- 使用可折叠的 SearchBar 在差异内容中搜索关键词
- 支持普通文本搜索和正则表达式搜索
- 搜索结果高亮显示，支持结果间跳转

## 13. 批量操作

### 13.1 功能概述

批量操作允许同时处理多条历史记录，大幅提高测试效率，适合大批量请求的处理场景。

### 13.2 多选

- 在历史记录面板中按住 **Ctrl** 键点击可多选非连续条目
- 按住 **Shift** 键点击可选择连续范围的条目
- 选中条目数以视觉反馈显示

### 13.3 批量重放

- 选中多条记录后右键选择 **"批量重放"**
- 插件依次重新发送所选请求（异步并发处理，不阻塞 UI）
- 重放结果自动追加到历史记录面板

### 13.4 批量越权测试

- 选中多条记录后右键选择 **"批量越权测试"**
- 自动遍历所有配置的用户会话进行越权检测
- 测试结果汇总到越权测试面板，方便统一分析

### 13.5 批量删除

- 选中多条记录后右键选择 **"删除"**
- 一键清理不需要的历史记录
- 删除操作会触发 GC 队列，自动清理关联的 Pool 数据

## 14. 报告导出

### 14.1 功能概述

报告导出模块使用 Template Method（模板方法）设计模式，将越权测试结果导出为多种格式的正式报告，方便交付、存档和团队分享。

### 14.2 导出操作

1. 在越权测试面板完成测试后，点击 **"导出报告"** 按钮
2. 选择导出格式：PDF / HTML / Markdown
3. 选择文件保存路径
4. 插件生成报告
5. HTML 格式报告生成后自动在浏览器中打开

### 14.3 报告格式

| 格式 | 说明 | 实现类 |
|------|------|--------|
| PDF | Apache PDFBox 3.0.1 原生生成，内嵌中文字体，适合正式交付 | PdfReportGenerator |
| HTML | FreeMarker 模板引擎渲染，样式美观，浏览器直接查看 | HtmlReportGenerator |
| Markdown | FreeMarker 模板生成纯文本报告，适合版本控制系统管理 | MarkdownReportGenerator |

### 14.4 报告内容

- **测试概要**：测试时间、目标范围、请求总数、越权发现数
- **会话统计**：各用户会话的测试结果统计汇总
- **逐端点详述**：每个测试端点的原始请求/响应和替换后请求/响应详情
- **cURL 命令**：每项测试的等效 cURL 命令（CurlBuilder 生成）
- **Postman 代码片段**：每项测试的 Postman 代码片段（PostmanSnippetBuilder 生成）

### 14.5 内容渲染

- 请求/响应体自动适配渲染（BodyRenderer）
- 二进制内容（图片、序列化数据等）自动转为 hex 或 base64 文本显示（BinaryContentRenderer）
- 文本内容保持原始格式和语法高亮

## 15. 高级技巧

### 15.1 HTTPS 请求处理

插件自动保留 HTTPS 协议信息：

- 从已保存请求重新发送时，HTTPS 协议不会丢失

- 全局 API 提取规则通过 YAML 文件共享

## 16. 常见问题

### Q: 插件加载后看不到 "Repeater Manager" 标签页？

A: 检查 Extender → Extensions 列表中插件是否加载成功，查看错误输出。确保使用的是 Burp Suite Professional 版本。

### Q: 重启 Burp Suite 后数据还在吗？

A: 在的。所有请求和历史记录都保存在 SQLite 数据库中，重启后自动加载。

### Q: 数据库文件在哪里？

A: 默认在 `~/.burp/` 目录下的会话子目录中。可在配置面板中查看当前会话路径。

### Q: 如何减少磁盘占用？

A: 1) 删除不需要的请求，GC 会自动清理关联数据；2) 使用 ERM 存档导出后清理旧会话。

### Q: ERM 加密安全吗？

A: ERM 加密使用 AES-256-CBC + HMAC-SHA256，密钥通过 PBKDF2 从密码派生。请使用强密码并妥善保管。

### Q: 支持哪些 Burp Suite 版本？

A: 仅支持 Burp Suite Professional。Community 版本不支持扩展加载功能。

### Q: 插件发送请求时走 Burp 的代理吗？

A: 默认不经过 Burp 代理，直接发送。如需调试，可在配置面板中启用 HTTP 代理并指向 Burp Proxy（如 127.0.0.1:8080）。
