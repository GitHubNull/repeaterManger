## 1. 安装与卸载

### 1.1 前置条件

- Burp Suite Professional 2024.1 或更高版本
- Java 17 或更高版本

### 1.2 安装步骤

1. 从 GitHub [Releases](../../releases) 页面下载最新版 JAR 文件
2. 打开 Burp Suite Professional
3. 转到 `Extensions` → `Installed` 选项卡
4. 点击 `Add` 按钮
5. 在 `Extension file` 中选择下载的 JAR 文件
6. 点击 `Next` 完成安装
7. 在 Burp Suite 输出面板中看到 **"Repeater Manager 插件加载成功"** 表示安装成功

> 首次加载后，插件会自动在 `~/.burp/` 目录下创建会话目录（以时间戳命名），内含数据库文件、Body 数据目录和日志目录。全局 API 提取规则存储在 `~/.burp/repeater_manager/api_extraction_rules.yaml`。

### 1.3 卸载

在 `Extensions` → `Installed` 中选中插件，点击 `Remove` 按钮即可卸载。已保存的数据不会被删除。

### 1.4 更新

卸载旧版本后，重新加载新版本的 JAR 文件即可。建议在更新前导出 ERM 存档备份数据。

## 2. 界面总览

插件加载后，Burp Suite 顶部会出现 **"Repeater Manager"** 标签页，包含三个子标签：

### 2.1 请求管理标签页

主工作界面，布局如下：

```
+---------------------------+-----------------------------------+
|                           |  [新建请求] [清空]    布局: [v]  |
|   请求列表                |                                   |
|   (搜索/过滤/颜色/备注)   |   请求编辑区    |    响应展示区   |
|                           |                                   |
+---------------------------+                                   |
|                           +-----------------------------------+
|   历史记录面板            |   状态栏: 响应大小 / 耗时 / 状态  |
|   (双击加载历史响应)      |                                   |
+---------------------------+-----------------------------------+
```

- **左侧上方**：请求列表面板，展示所有已接收/创建的请求
- **左侧下方**：历史记录面板，展示当前选中请求的重放历史
- **右侧**：请求编辑区和响应展示区，支持布局切换
- **底部**：状态栏，显示请求/响应的基本信息

### 2.2 配置标签页

包含多个子标签：

- **存储配置**：数据库路径、存储模式、自动保存参数
- **日志**：日志级别、文件日志、UI 日志、Burp 控制台日志
- **代理调试**：HTTP 代理配置
- **数据导入导出**：ERM 存档 / Postman Collection 导入导出
- **API 规则配置**：全局规则和项目规则的增删改查
- **越权测试**：方案 / 字段 / 用户会话 / 判断规则组 / 请求范围 / 重放配置 / 去重配置

### 2.3 日志标签页

显示插件运行日志，支持不同级别的日志过滤。

## 3. 请求管理

### 3.1 发送请求到插件

在 Burp Suite 的任何位置，右键点击 HTTP 请求，选择 **"发送到 Repeater Manager"**。支持的位置包括：

- Proxy → HTTP History
- Proxy → Intercept
- Intruder
- Scanner
- Repeater（原生）
- Spider

### 3.2 新建空白请求

点击右上角的 **"新建请求"** 按钮，将创建一个默认的 GET 请求模板：

```
GET / HTTP/1.1
Host: example.com
```

### 3.3 请求列表操作

- **选中请求**：单击列表中的请求，右侧加载请求内容和历史记录
- **颜色标记**：右键列表条目，选择颜色用于分类（如红色=高危、绿色=正常）
- **添加备注**：为请求添加文字说明，便于后续查找
- **搜索过滤**：在搜索框中输入关键词快速定位
- **列显示控制**：右键表头打开列控制对话框，自定义显示/隐藏表格列
- **删除请求**：右键选择删除，清理不需要的请求

## 4. 请求编辑与重放

### 4.1 编辑请求

右侧上方的请求编辑区提供语法高亮的文本编辑器，支持直接编辑：

- 请求行（方法、路径、协议版本）
- 请求头部
- 请求 Body

### 4.2 发送请求

编辑完成后，点击 **"发送"** 按钮（或使用快捷键），请求将通过 Burp Suite 发送。

- 请求在后台线程发送，不会冻结 UI
- 发送过程中光标变为等待状态
- 底部状态栏实时显示请求进度

### 4.3 查看响应

响应内容显示在右侧的响应展示区：

- 语法高亮显示 HTTP 响应
- 状态码、响应长度、响应时间等信息显示在状态栏
- 支持 HTTP/HTTPS 请求，协议信息自动保留

### 4.4 布局切换

右上角的下拉菜单提供四种布局模式：

| 布局 | 说明 |
|------|------|
| 左右布局 | 请求在左，响应在右（默认） |
| 上下布局 | 请求在上，响应在下 |
| 仅请求 | 只显示请求编辑区 |
| 仅响应 | 只显示响应展示区 |

### 4.5 清空内容

点击 **"清空"** 按钮可清空当前请求和响应内容。

## 5. 历史记录

### 5.1 自动记录

每次发送请求后，无论成功或失败，都会自动记录到历史面板：

- **成功记录**：包含完整的请求和响应数据
- **失败记录**：标记为"请求失败"，包含错误信息

### 5.2 查看历史

- 左下方面板显示当前选中请求的所有历史记录
- 记录按时间倒序排列（最新在最前）
- 每条记录显示：状态码、响应长度、响应时间、时间戳

### 5.3 回放历史

双击历史记录中的任意条目，将加载该次重放的：

- 原始请求数据（加载到请求编辑区）
- 响应数据（加载到响应展示区）
- 状态信息（更新到状态栏）

### 5.4 高级搜索

右键历史记录面板，选择 **"高级搜索"**：

- 支持多条件复合筛选（状态码、响应长度、时间范围等）
- 可快速定位特定历史记录

### 5.5 数据持久化

所有历史记录自动保存到 SQLite 数据库，即使重启 Burp Suite 也不会丢失。

## 7. 数据导入导出

### 7.1 ERM 存档格式

ERM（Repeater Manager）是插件专用的存档格式，包含完整的数据库和 Body 数据。

**导出**：
1. 进入 配置 → 数据导入导出
2. 点击 ERM 存档行的 **"导出"** 按钮
3. 选择是否加密（勾选"加密"复选框）
4. 如果加密，需输入密码（AES-256-CBC 加密 + HMAC-SHA256 完整性校验）
5. 选择保存路径，完成导出

**导入**：
1. 点击 ERM 存档行的 **"导入"** 按钮
2. 选择 .erm 文件
3. 如果文件已加密，需输入密码
4. 导入完成后数据自动加载

### 7.2 Postman Collection

支持 Postman Collection v2.1 格式的导入导出。

**导出**：
1. 点击 Postman 行的 **"导出"** 按钮
2. 选择保存路径，生成 .json 文件
3. 可直接在 Postman 中导入使用

**导入**：
1. 点击 Postman 行的 **"导入"** 按钮
2. 选择 Postman Collection v2.1 格式的 .json 文件
3. 导入完成后请求自动加载到列表

### 7.3 智能导入

点击 **"智能导入 (自动检测格式)"** 按钮，插件将自动检测文件格式：

- `.erm` 文件 → ERM 存档导入
- `.json` 文件 → Postman Collection 导入
- 其他格式 → 提示不支持

## 9. 存储与数据管理

### 9.1 会话目录结构

```
~/.burp/
├── repeater_manager_config.properties     # 插件配置文件
├── repeater_manager/
│   └── api_extraction_rules.yaml          # 全局 API 提取规则（跨会话共享）
└── session_20240101_120000/               # 会话目录（时间戳命名）
    ├── repeater_manager.sqlite3           # SQLite 数据库文件
    ├── blobs/                             # 外置 Body 数据目录
    └── logs/                              # 日志文件目录
```

### 9.2 Pool 去重机制

数据库采用 Pool 架构实现内容去重，减少存储空间占用：

- **string_pool**：存储域名、路径、查询参数等字符串，通过哈希去重
- **header_pool**：存储 HTTP 头部数据，相同头部只存一份
- **body_pool**：存储小体积 Body 数据（行内存储在数据库中）
- **file_pool**：存储大体积 Body 数据（外置文件存储在 blobs/ 目录中）

### 9.3 垃圾回收

- 后台 GC 服务自动清理零引用的 Pool 数据
- 默认每 10 分钟运行一次
- 删除请求后，相关 Pool 数据的引用计数减少
- 当引用计数为零时，数据进入 GC 队列等待清理
- 支持全量 ref_count 重算和立即触发 GC

### 9.4 自动保存

- 默认启用，每 5 分钟执行一次
- 执行数据库检查点操作，确保数据已写入主数据库文件
- 可在配置面板中调整保存间隔或禁用

## 10. API 规则提取

### 10.1 功能概述

API 规则提取引擎可自动从不规则的 HTTP 请求中提取标准化的 API 路径，便于整理和管理大量请求。例如从 `/api/v1/users/12345/detail` 中提取出 `/api/v1/users/{id}/detail`。

### 10.2 规则类型

- **全局规则**：存储在 `~/.burp/repeater_manager/api_extraction_rules.yaml`，跨会话共享，使用负数 ID
- **项目规则**：存储在会话 SQLite 数据库的 `api_extraction_rules` 表中，仅当前会话可用，使用正数 ID

### 10.3 提取源

| 提取源 | 说明 |
|--------|------|
| URL_PATH | 从 URL 路径中提取 |
| URL_QUERY | 从 URL 查询参数中提取 |
| HEADER | 从请求头部中提取 |
| BODY | 从请求 Body 中提取 |

### 10.4 提取方法

| 方法 | 说明 | 示例 |
|------|------|------|
| REGEX | 正则表达式匹配 | `^/api/(.*)$` |
| SUBSTR | 子串截取 | `start:end` 或 `start,length` |
| JSON_PATH | JSONPath 表达式 | `$.data.api_path` |
| XPATH | XPath 表达式 | `//api/@path` |

### 10.5 规则优先级

全局规则和项目规则按 `priority` 字段排序，采用 **first-match-wins** 策略，即第一个匹配的规则生效。

### 10.6 使用方法

1. 进入 **"配置"** → **"API 规则配置"** 标签页
2. 点击 **"添加规则"**，选择全局或项目规则
3. 配置规则名称、提取源、提取方法、匹配表达式、优先级
4. 保存规则
5. 在请求列表右键选择 **"重新提取 API"**，或规则变更后自动触发重提取

## 11. 越权测试

### 11.1 功能概述

越权测试模块提供自动化水平/垂直越权漏洞检测能力。通过配置多用户会话和判断规则，插件可自动拦截代理流量、替换身份凭证并重放请求，根据响应判断是否存在越权风险。

### 11.2 方案管理

方案（Scheme）是一组字段的组合，作为"字段"与"用户会话"之间的中间层。不同方案可对应不同的认证方式测试目标。

**方案概念**：
- 一个方案关联多个字段（如同时包含 Authorization Header 和 Cookie）
- 用户会话通过 `schemeId` 一对一关联方案
- 方案可持久化到全局（跨会话共享），启动时自动加载

**方案管理操作**：
- **新建方案**：配置名称、描述，选择关联的字段
- **编辑方案**：修改名称、描述、关联位置
- **删除方案**：删除方案（不影响字段本身）
- **启用/禁用**：控制方案是否可用
- **持久化到全局**：勾选后方案自动同步到全局 YAML 文件

**方案案例**：

| 方案名称 | 关联字段 | 适用场景 |
|----------|-------------|---------|
| Bearer 认证方案 | Authorization Header | 仅测试 Bearer 字段替换 |
| 混合认证方案 | Cookie + CSRF Token Header | 同时测试 Cookie 和 CSRF Token |
| API Key 方案 | X-API-Key Header + api_key URL参数 | 测试 API 密钥认证 |
| 游客方案 | （空列表，无字段） | 匿名用户专用，所有字段值为空 |

> 每个用户会话关联一个方案，会话的字段值按方案中的位置列表逐项填充。

### 11.3 字段配置

定义字段在 HTTP 请求中的具体位置。字段是方案和会话的基础。

**6 种位置类型**：

| 位置类型 | 说明 | 表达式示例 |
|----------|------|-----------|
| HEADER | HTTP 请求头（如 Authorization、Cookie） | `Authorization` 或 `Cookie` |
| JSON_BODY | JSON 请求体中的字段路径 | `$.data.token` |
| XML_BODY | XML 请求体中的节点路径 | `//session/token` |
| FORM_FIELD | x-www-form-urlencoded 表单字段 | `session_id` |
| MULTIPART_FIELD | multipart/form-data 表单字段 | `csrf_token` |
| URL_PARAM | URL 查询参数 | `token` 或 `access_token` |

**附加属性**：
- **持久化到全局**（`persistToGlobal`）：跨会话共享该位置定义
- **启用/禁用**（`enabled`）：控制该位置是否参与 字段替换
- 启动时自动加载全局字段到项目数据库（按 type+expression 去重）

### 11.4 判断规则组配置

> **v2.30.0 重要变更**：规则系统已从"多条件 AND/OR 组合"重构为**规则组（Rule Group）+ 单活跃规则集**模式。规则组内的条件按纯 AND 逻辑组合，全局同时只有一个规则组处于"活跃"状态。

**核心理念**：
- **规则组（Rule Group）**：一组条件的 AND 集合。组内所有已启用条件必须同时满足，规则组才算命中
- **单活跃规则集**：全局同时只有一个规则组处于"活跃"状态（`isActive=true`）。判决时仅使用活跃的规则组
- **兜底机制**：无活跃规则组，或活跃规则组内任一条件不满足 → 回退到默认相似度判决（`SIMILARITY >= 0.90`）

**判断目标与可用方法对照表**：

| 判断目标 | 可用判断方法 | 说明 |
|----------|------------|------|
| STATUS_CODE | EQUALS / NOT_EQUALS / NUMERIC_EQUALS / GREATER_THAN / LESS_THAN | 状态码数值比较 |
| RESPONSE_BODY | CONTAINS / NOT_CONTAINS / REGEX / EQUALS / NOT_EQUALS / LENGTH_DIFF | 响应体匹配；LENGTH_DIFF 比较与基准响应的字节长度差 |
| RESPONSE_HEADER | CONTAINS / NOT_CONTAINS / REGEX | 响应头匹配 |
| RESPONSE_TIME | GREATER_THAN / LESS_THAN / NUMERIC_EQUALS | 响应时间判断（毫秒） |
| SIMILARITY | GREATER_THAN / LESS_THAN / NUMERIC_EQUALS | 响应相似度（0.0~1.0，系统默认内置规则：SIMILARITY >= 0.90） |

**条件运算符**：

| 运算符 | 含义 | 说明 |
|--------|------|------|
| AND（且） | 当前条件与之前累积结果都为真，继续求值 | 默认运算符 |
| OR（或） | 当前条件或之前累积结果为真即继续 | 同一规则组内灵活组合 |
| NOT（取反） | 翻转当前条件的匹配结果 | 通过条件上的"取反"复选框控制 |

> **求值顺序**：从左到右按 `sort_order` 顺序求值。第一条条件的运算符作为占位符，实际求值时与初始值 `true` 组合。

#### 11.4.1 规则配置案例

以下 8 个案例从简单到复杂，每个案例对应一个规则组。在规则组模式下，组内条件按 AND 逻辑组合（运算符可设为 OR 实现灵活组合）。

**案例 1：状态码异常检测（单条件）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | NOT_EQUALS | 200 |

> **场景**：最基础的越权检测。当普通用户以管理员身份重放请求时，如果服务端正确鉴权，应返回 403/401 等非 200 状态码；若返回 200，则判断为越权。

**案例 2：敏感关键词检测（单条件）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | RESPONSE_BODY | CONTAINS | "admin" |

> **场景**：检测响应体中是否包含高权限关键词。适用于接口返回数据中包含"admin"、"root"、"administrator"等敏感字段的场景。

**案例 3：精准水平越权检测（双条件 AND）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | — | RESPONSE_BODY | CONTAINS | "admin" |

> **场景**：状态码正常 AND 响应体包含越权数据 → 精准越权。两条条件必须同时满足才告警，避免单独一条条件导致的误报。

**案例 4：排除正常拒绝页面（AND + NOT）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | ☑ | RESPONSE_BODY | CONTAINS | "请先登录" |

> **场景**：通过 NOT 排除正常的拒绝页面。有些服务会在 200 状态下返回"请先登录"文本（前端路由拦截），这种情况下不应判定为越权。AND + NOT 确保：状态码是 200 且响应体不含"请先登录"才告警。

**案例 5：多关键词匹配（AND + OR）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | — | RESPONSE_BODY | CONTAINS | "admin" |
| OR | — | RESPONSE_BODY | REGEX | "ssn\\|password\\|credit_card" |

> **场景**：状态码正常 AND（包含"admin" OR 匹配敏感数据正则）。任一关键词命中即告警，适用于需要检测多种敏感数据类型的场景。

**案例 6：数据泄露检测（双条件 AND）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | RESPONSE_BODY | REGEX | "email.*phone" |
| AND | — | RESPONSE_BODY | LENGTH_DIFF | 1024 |

> **场景**：出现敏感字段组合（email+phone）AND 响应体显著变大（超过 1024 字节）。LENGTH_DIFF 比较当前响应体与基准响应体的字节长度差，适用于检测大量数据泄露。

**案例 7：相似度兜底（默认内置规则）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | SIMILARITY | GREATER_THAN | 0.90 |

> **场景**：系统自动创建的默认兜底规则组，优先级最低。当用户没有设置活跃规则组，或活跃规则组不匹配时使用。相似度 >= 90% 表示与基准响应高度相似，判定为越权。用户可禁用、修改或删除此规则组。

**案例 8：访客未授权检测（规则组 + NOT，v2.31.0 新增）**

| 运算符 | NOT | 目标 | 方法 | 表达式 |
|--------|-----|------|------|--------|
| — | — | STATUS_CODE | EQUALS | 200 |
| AND | ☑ | RESPONSE_BODY | CONTAINS | "请先登录" |

> **场景**：使用匿名用户（所有字段值为空）访问需要认证的接口。服务端正确拦截应返回 401/403 或含"请先登录"的页面；若直接返回 200 且无拒绝文案，说明存在未授权访问漏洞。此案例与案例 4 配置相同，但结合匿名用户使用。

#### 11.4.2 规则组活跃状态管理

- **设置活跃规则组**：在判断规则表格中勾选目标规则组的"活跃"列（☑）
- **全局唯一性**：设置新的活跃规则组时，系统自动取消其他规则组的活跃状态
- **UI 视觉反馈**：活跃规则组在表格中以特殊样式高亮显示
- **无活跃规则组时**：判决自动回退到默认相似度判决（`SIMILARITY >= 0.90`）
- **多规则组切换**：适合不同测试场景快速切换（如切换"严格检测"和"宽松检测"规则组）

#### 11.4.3 规则复用

- 规则组可设置为**全局规则**（`global=true`），跨项目共享
- 支持 **YAML 导出/导入**，实现规则组的跨会话复用和团队共享（导入时自动去重）
- 建议沉淀典型越权场景的规则组集合，后续针对相同或相似目标直接导入使用

### 11.9 请求范围配置

指定需要测试的 URL 模式，只有匹配范围的请求才会被拦截和测试。

### 11.6 匿名用户

**功能概述**：一键创建所有字段值为空的访客用户，用于未授权访问测试。

**操作流程**：
1. 进入 **"配置"** → **"越权测试"** → **"用户会话"** 标签页
2. 点击 **"添加匿名用户"** 按钮
3. 系统自动进行智能方案匹配：
   - **优先级 1**：复用已有用户的方案（如果已有用户关联了方案）
   - **优先级 2**：自动匹配唯一启用的方案
   - **优先级 3**：多方案时弹出选择对话框，由用户手动选择

**空字段值的语义**：匿名用户的所有字段值为空字符串，请求重放时：
- **HEADER** 类型：删除该 Header（而非设置空值）
- **JSON_BODY** 类型：移除 JSON 属性
- **XML_BODY** 类型：移除 XML 节点
- **FORM_FIELD / MULTIPART_FIELD** 类型：移除表单字段
- **URL_PARAM** 类型：移除 URL 查询参数

> 这种"移除"语义模拟了请求中不存在认证参数的未登录状态，比设置空字符串更接近真实的未授权访问场景。

**适用场景**：
- 未授权访问测试（检测是否需要登录才能访问）
- 游客权限边界测试（匿名用户能否看到本应隐藏的数据）
- 接口认证缺失检测

### 11.7 去重配置

**功能概述**：避免同一 API 被重复越权测试，提高自动化测试效率。

**核心概念**：
- **去重策略（DedupStrategy）**：定义从请求中提取去重键的方式（6 种）
- **保留策略（DedupKeepPolicy）**：定义重复请求的保留规则（3 种）
- **优先级链式匹配**：按优先级从高到低遍历所有启用的配置，首个成功提取到去重键的立即返回

**6 种去重策略**：

| 策略 | 说明 | 表达式示例 |
|------|------|-----------|
| PATH | 按 URL 路径去重 | （无需表达式） |
| API | 按提取后的 API 路径去重 | （无需表达式） |
| JSON_BODY_FIELD | 按 JSON Body 中指定字段的值去重 | `user_id` |
| XML_BODY_FIELD | 按 XML Body 中指定节点的值去重 | `//user/id` |
| FORM_FIELD | 按表单字段值去重 | `order_id` |
| URL_PARAM | 按 URL 参数值去重 | `page` |

**3 种保留策略**：

| 策略 | 说明 |
|------|------|
| FIRST | 保留第一个匹配的请求，后续重复请求跳过 |
| LAST | 保留最后一个匹配的请求，之前的被覆盖 |
| MIDDLE | 保留中间位置的请求（取索引中位数） |

**存储方式**：
- **全局持久化**：配置存储在 `~/.burp/repeater_manager/dedup_configs.yaml`，跨会话共享
- **会话级临时**：仅当前会话有效，重启后清除

### 11.8 会话解析

**功能概述**：从剪贴板中的 HTTP 报文自动解析出用户会话的字段值和位置，大幅简化会话配置流程。

**支持格式**：
1. **原始 HTTP 报文**：标准的 HTTP 请求报文（如 Burp Proxy 中复制的请求）
2. **Chrome "Copy as fetch"**：Chrome DevTools Network 面板的 fetch 复制格式
3. **Chrome "Copy as fetch (Node.js)"**：Node.js 兼容的 fetch 复制格式

**操作流程**：
1. 在 Burp Proxy / Chrome DevTools 中复制目标请求
2. 进入 **"配置"** → **"越权测试"** → **"用户会话"** 标签页
3. 点击 **"从剪贴板解析"** 按钮
4. 插件自动检测剪贴板内容格式并转换为原始 HTTP 报文
5. 自动提取字段值和位置信息
6. 选择目标方案（无匹配方案时弹出选择对话框）
7. 确认后创建或更新用户会话

> Chrome fetch 格式支持单双引号、转义序列、嵌套对象等复杂 JS 语法解析。

### 11.10 执行测试

1. 完成上述配置后，在越权测试面板中设置一个**活跃规则组**（勾选目标规则组的"活跃"列）
2. 开启 **"自动检测"** 开关
3. 插件拦截匹配范围的代理流量
4. 自动替换字段值后重放请求（匿名用户的空字段值执行"移除"操作）
5. 根据活跃规则组评估越权风险（规则组内所有条件 AND 满足 → 越权；任一条件不满足 → 回退相似度判决）
6. 在 **"越权测试"** 面板查看结果：
   - **红色**：疑似越权，需人工确认
   - **绿色**：安全，未检测到越权

> **判决流程（三层）**：① 基准响应无效 → 标记"错误"；② 有活跃规则组且全部条件满足 → 标记"越权"；③ 无活跃规则组或不匹配 → 使用相似度兜底判决（≥ 阈值 → 越权，< 阈值 → 安全）。

> **相似度判断说明**：系统首次启动时自动创建一条"默认相似度规则组"（优先级最低），条件为 `SIMILARITY >= 0.90`。当没有活跃规则组或活跃规则组不匹配时，该兜底规则组生效。用户可在"判断规则"标签页中查看、修改、设为活跃、禁用或删除此规则组。

## 12. 报文比对

### 12.1 功能概述

报文比对模块提供请求/响应对的差异比较能力，支持在越权测试场景中对比原始请求和替换 Token 后的请求及相应响应。也支持对比同一请求的任意两条历史记录。

### 12.2 启动比对

- 选择两条历史记录后右键，可直接对比两者的请求和响应

### 12.3 差异显示

- **绿色**：新增内容（added）
- **红色**：删除内容（deleted）
- **黄色**：修改内容（modified）
- 支持字符级行内差异高亮，DiffEngine 基于 LCS（最长公共子序列）算法变体实现

### 12.4 同步滚动

- 原始和替换两端面板通过 SynchronizedScrollPanel 自动联动滚动
- 对比大报文时保持上下文一致，不会出现查看错位

### 12.5 差异导航

- 使用 DiffNavigator 提供的 **上一处差异** / **下一处差异** 按钮快速跳转
- 当前差异区域自动高亮显示
- 顶部显示差异总数和当前位置

### 12.6 搜索

- 使用可折叠的 SearchBar 在差异内容中搜索关键词
- 支持普通文本搜索和正则表达式搜索
- 搜索结果高亮显示，支持结果间跳转

## 14. 报告导出

### 14.1 功能概述

报告导出模块使用 Template Method（模板方法）设计模式，将越权测试结果导出为多种格式的正式报告，方便交付、存档和团队分享。报告生成基于统一的数据模型（`ReportData`），由 `ReportExporter` 调度，支持明文和加密两种导出模式。

### 14.2 导出操作

1. 在越权测试面板完成测试后，点击 **"导出报告"** 按钮
2. 选择导出格式：PDF / HTML / Markdown / ERMR（加密压缩包）
3. 选择文件保存路径
4. 若选择 ERMR 加密模式，需输入密码（AES-256-CBC 加密）
5. 插件生成报告
6. HTML 格式报告生成后自动在浏览器中打开

### 14.3 报告格式

| 格式 | 扩展名 | 说明 | 实现类 |
|------|--------|------|--------|
| PDF | `.pdf` | Apache PDFBox 3.0.1 原生生成，内嵌中文字体，适合正式交付 | `PdfReportGenerator` |
| HTML | `.html` | FreeMarker 模板引擎渲染，样式美观，浏览器直接查看 | `HtmlReportGenerator` |
| Markdown | `.md` | FreeMarker 模板生成纯文本报告，适合版本控制系统管理 | `MarkdownReportGenerator` |
| ERMR（加密容器） | `.ermr` | 将任意格式报告打包为 AES-256-CBC + HMAC-SHA256 加密归档，适合安全传输 | `ReportExporter`（调度器） |

> ERMR 容器模式可叠加在 PDF/HTML/Markdown 之上，生成 `.ermr` 加密文件，确保敏感报告数据在传输和存储过程中的安全性。

### 14.4 报告内容

**测试概要**：
- 测试时间、目标范围、请求总数、越权发现数
- 基准用户名称和配置的活跃规则组名称

**会话统计**：
- 各用户会话的测试结果统计汇总
- 判决结果使用中文显示名：

| 枚举值 | 报告显示名 | 含义 |
|--------|-----------|------|
| `ESCALATED` | ⚠ 越权 | 检测到越权风险，需人工确认 |
| `NOT_ESCALATED` | ✔ 安全 | 未检测到越权 |
| `ERROR` | ✗ 错误 | 判决过程出错（如基准响应无效） |
| `PENDING` | 待判定 | 尚未完成判决（如状态码差异需人工确认） |

> 报告模板通过 `JudgmentResult.toDisplayName()` 方法将枚举值统一转换为中文显示名，确保报告中不出现 `"ESCALATED"` 等原始枚举字符串。

**逐端点详述**（每个测试 API 端点包含）：
- 原始请求/响应和替换 Token 后的请求/响应详情
- 匹配的**规则组名称**（`matchedRuleName`）：当判决由规则组命中时，报告中明确展示命中的规则组名称，方便追溯判决依据
- 相似度得分（`similarity`）：原始响应与替换后响应的相似度百分比
- HTTP 状态码、响应长度、响应时间

**复现命令**：
- **cURL 命令**：每项测试的等效 cURL 命令（`CurlBuilder` 生成）
- **Postman 代码片段**：每项测试的 Postman 代码片段（`PostmanSnippetBuilder` 生成），可直接导入 Postman 复现

### 14.5 内容渲染

- 请求/响应体自动适配渲染（`BodyRenderer`）
- 二进制内容（图片、序列化数据等）自动转为 hex 或 base64 文本显示（`BinaryContentRenderer`）
- 文本内容保持原始格式和语法高亮
- PDF 报告中超长文本（如 Base64 字符串）自动截断，并提示查看 HTML 报告获取完整数据

### 14.6 敏感数据警告

导出报告时，插件会弹出敏感数据警告对话框，提示报告将包含完整的请求/响应数据（可能包括 Bearer Token、Session Cookie 等敏感信息）。建议：
- 使用 ERMR 加密容器模式保护报告数据
- 交付前审查报告中是否包含未脱敏的认证信息
- 使用强密码加密 ERMR 文件

## 16. 常见问题

### Q: 插件加载后看不到 "Repeater Manager" 标签页？

A: 检查 Extender → Extensions 列表中插件是否加载成功，查看错误输出。确保使用的是 Burp Suite Professional 版本。

### Q: 重启 Burp Suite 后数据还在吗？

A: 在的。所有请求和历史记录都保存在 SQLite 数据库中，重启后自动加载。

### Q: 数据库文件在哪里？

A: 默认在 `~/.burp/` 目录下的会话子目录中。可在配置面板中查看当前会话路径。

### Q: 如何减少磁盘占用？

A: 1) 删除不需要的请求，GC 会自动清理关联数据；2) 使用 ERM 存档导出后清理旧会话。

### Q: ERM 加密安全吗？

A: ERM 加密使用 AES-256-CBC + HMAC-SHA256，密钥通过 PBKDF2 从密码派生。请使用强密码并妥善保管。

### Q: 支持哪些 Burp Suite 版本？

A: 仅支持 Burp Suite Professional。Community 版本不支持扩展加载功能。

### Q: 插件发送请求时走 Burp 的代理吗？

A: 默认不经过 Burp 代理，直接发送。如需调试，可在配置面板中启用 HTTP 代理并指向 Burp Proxy（如 127.0.0.1:8080）。
